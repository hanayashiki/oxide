# `as` cast operator

## Requirements

`as` is the only source-level mechanism for type conversion: `expr as
T`. It already exists at every layer (parser/HIR/codegen) but has had
no typeck-side validation — typeck accepts every shape, and codegen
panics when it can't lower the result.

This spec adds **typeck-side cast validation**: a fixed allowed-set
of source-target pairs. Anything off the table is a structured error
(`E0264 InvalidCast`), replacing what is today an unrecoverable
compiler crash.

The work is almost entirely in typeck. Codegen gains one pointer-to-
pointer arm (a no-op, since LLVM `ptr` is type-agnostic). AST and
HIR are unchanged — `ExprKind::Cast { expr, ty }` was always there.

## Subset-of-Rust constraint

Anything we accept must also parse-and-mean-the-same-thing in Rust.
Where we differ, we accept *fewer* programs. We do not invent new
cast shapes Rust doesn't have.

The notable divergence from Rust: we **disallow** `*const T as *mut
T` and pointer reinterpretation (`*const T as *const U`). Rust
allows both via `as` because raw pointers in Rust carry no enforced
borrow contract — the `unsafe { ... }` block at the deref site is
where actual harm gets gated. We have no `unsafe` block to gate
deref, so we gate the cast instead. See "Why we diverge from Rust
on raw-pointer casts" below for the full reasoning.

## Acceptance

```rust
fn truncate(x: u32) -> u8 { x as u8 }                    // ✓ int→int
fn widen(b: bool) -> i32 { b as i32 }                    // ✓ bool→int
fn drop_mut(p: *mut i32) -> *const i32 { p as *const i32 }  // ✓ mut→const same pointee
```

```rust
fn launder(p: *const i32) -> *mut i32 { p as *mut i32 }  // ✗ E0264
fn nonsense(p: Point) -> u8 { p as u8 }                  // ✗ E0264
fn intbool(x: i32) -> bool { x as bool }                 // ✗ E0264 — write `x != 0`
fn reinterpret(p: *const u8) -> *const i32 { p as *const i32 }  // ✗ E0264
```

The four error cases each compile-crash today; the spec replaces the
crashes with structured diagnostics.

## Position in the pipeline

```
Source ─▶ tokens ─▶ AST ─▶ HIR ─▶ typeck (this spec adds the check) ─▶ codegen
```

AST and HIR shapes (`ExprKind::Cast { expr, ty }` /
`HirExprKind::Cast { expr, ty }`) are unchanged.

## Typeck rules

`infer_cast(expr, target_hir_ty) -> TyId`:

1. `src_ty := infer_expr(expr)`, then resolve through Infer.
2. `dst_ty := resolve_named_ty(target_hir_ty)`.
3. Validate via `cast_kind(src_ty, dst_ty)` — see table below. On
   `Reject`, push `TypeError::InvalidCast { src, dst, span }` and
   return `dst_ty` anyway (so cascades stay typed).
4. Return `dst_ty`.

The validation function returns one of:

```rust
enum CastKind {
    IntToInt,           // both Prim, both integer (incl. width/signedness change)
    BoolToInt,          // src Prim::Bool, dst Prim integer
    PtrToPtr,           // both Ptr, same pointee shape, dst-mut ≥ src-mut
    Identity,           // src == dst (no-op, but legal — e.g. `x as i32` where x: i32)
    Reject,             // anything else; error
}
```

### Allowed set (full table)

For all combinations after Infer-resolution:

| Source `TyKind` | Target `TyKind` | Result | Notes |
|---|---|---|---|
| `Prim(p)` integer | `Prim(q)` integer | `IntToInt` | Standard widening/narrowing/signedness change |
| `Prim(Bool)` | `Prim(_)` integer | `BoolToInt` | i1 → iN via zext |
| `Ptr(t1, m1)` | `Ptr(t2, m2)` | `PtrToPtr` (only if `t1 == t2` *and* `mut_le(m1, m2)`) | Same pointee, `mut→const` OK, `const→mut` Reject |
| any | same `TyId` | `Identity` | Trivial; result equals source |
| `Error` | any | `Identity` | Poison absorbs |
| any | `Error` | `Identity` | Poison absorbs |
| anything else | anything else | `Reject` | Below |

### Rejected cases (each emits E0264)

- **Integer ↔ Bool** (`x as bool`): meaningless; force `x != 0`. Rust
  also rejects.
- **Adt / Fn / Unit / Never on either side** (`Point as u8`,
  `1 as Point`, `f as i32` where `f` is a fn): cross-category
  conversions are nonsense. Rust also rejects.
- **`*const T as *mut T`**: silently grants write access. Rust allows
  (its unsafety boundary is at deref); we have no such boundary, so
  we gate the cast.
- **`*const T as *const U`** / **`*mut T as *mut U`** where T ≠ U
  (different pointee shape): pointer reinterpretation. Rust allows;
  we forbid until an FFI use case demands it.
- **Pointer ↔ integer**: deferred. Rust allows (`p as usize`,
  `0 as *const T`); useful for null-pointer-creation and
  address-printing, but nothing on our roadmap needs it yet.

### `mut_le` recap

The mutability subtype rule from `07_POINTER.md` §3, reused here:

```text
mut_le(actual, expected) iff
    actual = Const  AND  expected = Const
 OR actual = Mut    AND  expected = Const
 OR actual = Mut    AND  expected = Mut
```

`Const → Mut` is the one rejected combination.

## Why we diverge from Rust on raw-pointer casts

Rust's model: **raw pointers carry no enforced contract.** The
`*const`/`*mut` tags are documentation; they don't gate operations.
The actual unsafety lives at deref/write — and Rust forces those into
`unsafe { }` blocks, where the user explicitly takes responsibility.
So `*const T as *mut T` is "harmless paperwork"; the real check fires
when you write through `q`.

Our model: **mutability is enforced wherever it appears.** We have no
`unsafe` block — there's nowhere for the user to opt into "yes I know
what I'm doing." If we allowed `*const T as *mut T`, the resulting
`*mut T` would be silently writable downstream with no marker that
the original pointer wasn't write-licensed.

Consequence: we tighten `as`. The two real Rust use cases for
mut-upgrade casts in safe code are:

- **Rust APIs that historically returned `*const` and now want
  `*mut`** — handled by `<*const T>::cast_mut()`. We don't have
  this method-on-pointer story; no equivalent need.
- **C interop where the C signature is `T*` (which is read-write
  in C) but Rust's binding chose `*const T` for caution** — the
  user "fixes" by re-casting. For us: declare the extern with the
  right mutability up front, and the issue disappears.

We're sacrificing genuinely-zero programs by being stricter.

Pointer reinterpretation (`*const u8 as *const i32`) is the same
shape of decision: Rust allows the cast and gates deref; we gate
the cast. Same justification; same future escape hatch (`unsafe
{ }` once we have it).

## Codegen

The existing `emit_cast` handles `IntToInt` and `BoolToInt` via the
`into_int_value()` + width-compare path. **One new arm needed**:
`PtrToPtr` returns the pointer value untouched.

```rust
fn emit_cast(&self, fx, eid, inner) -> BasicValueEnum {
    let dst_ty = self.ty_of(eid);
    let src_ty = self.ty_of(inner);
    let v = self.emit_expr(fx, inner).expect("cast operand produced no value");

    match (
        self.typeck_results.tys().kind(src_ty),
        self.typeck_results.tys().kind(dst_ty),
    ) {
        (TyKind::Ptr(..), TyKind::Ptr(..)) => v,                  // no-op
        (TyKind::Prim(_), TyKind::Prim(_)) => self.emit_int_cast(v.into_int_value(), src_ty, dst_ty),
        _ => panic!("v0 codegen: invalid cast — typeck should have rejected"),
    }
}
```

The Identity arm (`x as i32` where `x: i32`) falls through naturally
— either the int-to-int path returns `v` unchanged (same width), or
the ptr-to-ptr path returns `v` unchanged.

LLVM `ptr` is type-agnostic so no `bitcast` is needed for the
pointer arm. The mutability tag was a typeck concept only.

## Errors

```rust
pub enum TypeError {
    ...
    /// `expr as Ty` where the source-target pair isn't in the allowed
    /// cast table. See spec/12_AS.md "Allowed set". E0264.
    InvalidCast { src: TyId, dst: TyId, span: Span },
}
```

Diagnostic message:

```
[E0264] Error: cannot cast `<src>` as `<dst>`
   ╭─[ ... ]
   │
 N │     <expr> as <Ty>
   │     ──────┬──────  
   │           ╰──────── invalid cast
   │ 
   │ Help: <op-specific>
───╯
```

Help text varies by case to give the user a path forward:

- `*const T as *mut T` →
  *"raw `*const` → `*mut` is not allowed; declare the source as
  `*mut T` from the start, or pass the value through an `extern`
  signature that takes `*mut T`."*
- Integer → bool → *"compare against zero: `x != 0`."*
- Adt cast → *"struct values cannot be cast to other types; access a
  field or write a conversion fn."*
- Pointer reinterpretation →
  *"different pointee types are not interchangeable in v0; bind the
  pointer to a local typed `*const U` directly via the source-of-truth
  declaration, or wait for the future `as` extension."*
- `expr as Ty` where the dst type doesn't exist (already handled by
  `UnknownType` from `resolve_named_ty`) — no E0264 needed.

`MutateImmutable` already lives at **E0263** (spec/10_ADDRESS_OF.md),
so `InvalidCast` claims the next free typeck code, **E0264**, with no
renumbering.

## Worked examples

### Allowed: integer narrowing

```rust
fn lo(x: u32) -> u8 { x as u8 }
```

`cast_kind(u32, u8)` = `IntToInt`. Codegen emits `trunc i32 %x to i8`.

### Allowed: bool to int

```rust
fn b(b: bool) -> i32 { b as i32 }
```

`cast_kind(bool, i32)` = `BoolToInt`. Codegen emits
`zext i1 %b to i32`.

### Allowed: outer mutability drop

```rust
fn ro(p: *mut i32) -> *const i32 { p as *const i32 }
```

`cast_kind(*mut i32, *const i32)` = `PtrToPtr` (same pointee, `Mut →
Const`). Codegen returns the pointer value unchanged; LLVM `ptr` is
mutability-agnostic.

### Rejected: mutability upgrade

```rust
fn rw(p: *const i32) -> *mut i32 { p as *mut i32 }
//                                ─────────┬─────────
//                                         ╰── E0264 InvalidCast
//   help: raw `*const` → `*mut` is not allowed; ...
```

### Rejected: struct to int

```rust
struct Point { x: i32, y: i32 }

fn xy(p: Point) -> u8 { p as u8 }
//                       ─┬─
//                        ╰── E0264 InvalidCast
//   help: struct values cannot be cast to other types; ...
```

## Out of scope (this round)

- **Pointer ↔ integer** (`p as usize`, `0 as *const T`). Useful but
  not on the immediate roadmap. The acceptance program for the
  socket-server (`example-projects/socket-server/`) doesn't need it
  — the kernel API is happy with the typed-pointer extern signatures
  we already use. When a use case lands (e.g., `null` for `signal()`
  or `ioctl`), add the integer-to-pointer arm with a literal-only
  restriction first (only `0 as *const T`) before generalizing.
- **Pointer reinterpretation** (`*const T as *const U`, T ≠ U).
  Genuinely needed for some FFI patterns (treating a byte buffer
  as a typed struct). Defer until we have an `unsafe { ... }` block
  to gate the deref consequence.
- **Float casts.** No floats in the language yet.
- **`char` casts.** `char` is `u8` per the existing model, so
  `c as u8` is `IntToInt`. No special case.
- **`as` chains.** `e as T as U` already parses left-associative
  via the existing Pratt level; each step runs `cast_kind`
  independently. Nothing new.

## Out of scope (forever-ish)

- Implicit numeric conversions. Oxide stays Rust-strict: `let x: i64
  = 1i32` is an error; the user writes `1i32 as i64`. We do not add
  silent widening.
- Float-to-bool, bool-to-float (when floats arrive). Same shape of
  rejection as int-to-bool today.
- Trait objects / fat pointers. We have neither.

## Note on existing code

Today's codegen `emit_cast` (lower.rs:754) uses `into_int_value()`
unconditionally on the operand. After this spec, the entry to
`emit_cast` dispatches on `(src_kind, dst_kind)` and only takes the
int path when both are primitives. The struct/ptr/etc. cases are
unreachable post-typeck (rejected as `InvalidCast`), but the
codegen panic message is updated to "typeck should have rejected"
to make the invariant clear.
